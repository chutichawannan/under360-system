# 🚚 พี่เก่ง — Delivery Agent (a0.2-spec · DRAFT)

## 1. Purpose
Codify การจัดรอบส่ง + เลือกวิธีส่งที่ถูกที่สุดต่อออเดอร์ — **เป้าหมายตรง = ลดค่าส่งที่ร้าน subsidize อยู่ 30,000–50,000฿/เดือน** (ร้านเก็บลูกค้าไม่เกิน ~200฿ แต่จ่ายจริงมากกว่า ส่วนเกินร้านกลืน) · แทนงานที่ตอนนี้แอดมินจัดรอบ/เรียกรถเอง

## 2. Trigger (dual-mode — ตาม CLAUDE.md)
- **Real-time monitor** — ออเดอร์เข้า → ประเมินทันที (ลูกค้าต่อรองเวลาได้ ≤2 ครั้ง/เดือน)
- **Route planner** — ทุกวัน ~17:00 รวมออเดอร์ทั้งวันวางแผนเส้นทาง/จับกลุ่มรอบเดียวกัน

## 3. Inputs
| ข้อมูล | สถานะ |
|--------|-------|
| ออเดอร์วันนี้ + ที่อยู่/พิกัดผู้รับ | 🟢 `orders` (มี delivery_address · พิกัดยังไม่แน่ใจว่ามี lat/lng — ดูข้อ ❓) |
| ครัวต้นทาง 13.7179969, 100.5010971 | 🟢 CLAUDE.md |
| ช่วงเวลาที่ลูกค้าสะดวกรับ (เช้า <12:00 / บ่าย 13–17:00 / ยืดหยุ่นทั้งวัน) | 🟡 ต้องเช็คว่า order เก็บ field นี้หรือยัง |
| Lalamove v3 quote (realtime price ตามระยะ) | 🔴 ต่อ API |
| ต้นทุนแมสร้านเอง (ต่อรอบ/ต่อกม.) | 🔴 ต้องมีสูตร (ดู ❓) |

## 4. Logic (ร่าง — รอ verify ตัวเลขจริง)
**ชั้น 1 — จับกลุ่มออเดอร์ (batching):** ออเดอร์ที่อยู่โซน/ทิศทางเดียวกัน + ช่วงเวลารับทับกัน → รวมรอบเดียว (ลดจำนวนเที่ยว = ลดต้นทุนหลัก)
**ชั้น 2 — เลือกวิธีส่งถูกสุดต่อกลุ่ม:** เทียบ (ก) แมสร้านเอง (ข) Lalamove quote (ค) [Grab/อื่น?] → เลือกถูกสุดที่ยังส่งทันเวลา
**ชั้น 3 — จูงใจ slot ยืดหยุ่น:** ออเดอร์ที่เลือก "ยืดหยุ่นทั้งวัน" = จัดเข้ารอบที่รถว่าง/คุ้มสุด (มีส่วนลดล่อ — ตาม CLAUDE.md)
**ชั้น 4 — flag ออเดอร์ที่ค่าส่งจริง > เพดานเก็บลูกค้ามาก** → เตือนแอดมิน (ตัวที่กิน subsidize หนัก)

## 5. Output
- **รายการจัดรอบ/วัน** ให้แอดมิน confirm — แต่ละรอบ: กลุ่มออเดอร์ + วิธีส่งแนะนำ + ค่าส่งประเมิน + เวลาออกรถ
- **สรุปเงิน:** วันนี้ subsidize เท่าไหร่ / ประหยัดได้เท่าไหร่จากการ batch (ตัววัด KPI ของ agent นี้ตรงๆ)
- (future) หน้าจัดแมสใหม่แบบ "รายการ" — แมสอัพเดทสถานะจากมือถือ + ถ่ายรูป proof of delivery + LINE แจ้งลูกค้า (ตอนนี้หน้าแมสเซนเจอร์แค่ดู ไม่มีปุ่ม)

## 6. Human-in-loop (trust-building)
- ช่วงแรก: แอดมิน approve ทุกรอบก่อนเรียกรถ · Lalamove auto-booking = ปิดไว้ก่อน เปิดเมื่อเชื่อใจ

## 7. Tables
- 🟢 `orders` (เดิม) · 🔴 อาจต้อง `delivery_routes` / `delivery_legs` (เก็บกลุ่มรอบ + วิธีส่ง + ค่าส่งจริง) · 🔴 proof of delivery (รูป + timestamp) — future

## 8. Dependencies
- ✅ ไม่ block น้องนิว (คนละสาย) — ทำขนานได้
- 🔴 ต้องมี **พิกัด lat/lng ของ order** ถึงจะ batch/route ได้ (ถ้าตอนนี้เก็บแค่ text address → ต้อง geocode ด้วย Google Maps API ที่มี key แล้ว)
- 🔴 Lalamove v3 API credentials

## 9. v1 scope vs future
- **v1 (ทำก่อน):** ชั้น 1+2+4 (batch + เทียบราคา + flag) → แค่นี้ก็เริ่มลด subsidize ได้ · แอดมิน approve มือ · ยังไม่ต้อง auto-booking
- **future:** auto-booking Lalamove · หน้าแมสมือถือ + proof of delivery + LINE แจ้งลูกค้า · ชั้น 3 ส่วนลด slot

## 10. ✅ คำตอบจากนัท (14 ก.ค.) + ที่ยังค้าง
1. **ส่งด้วยอะไร:** ตอนนี้ **Lalamove 100%** (messenger เพิ่งลาออก) · ถ้าจ้างแมสประจำ = เงินเดือน ~17,000฿ แต่ต่อให้จ้างก็ยังต้องเรียก Lalamove เสริมอยู่ดี → logic ต้องเทียบ "แมสประจำ (ถ้ามี) vs Lalamove" เป็นหลัก
2. **ค่าส่ง — เก็บลูกค้า (สูตรจริงในโค้ด `liff_customer.html` `calcDeliveryFee`):** เริ่ม 10฿ + ตามระยะ **0-5km ฿4/km · 5-10km ฿5/km · 10-15km ฿7/km · 15-30km ฿9/km · >30km ฿9/km** · slot "รับได้ตลอดวัน" ลด 10฿ · **จ่ายจริง Lalamove = ยังไม่รู้** (โค้ดตอนนี้เป็น placeholder `30 + 8฿/km`, >30km flat 200฿ — ยังไม่ต่อ API จริง)
   - 🎯 **งานวัด subsidy จริง:** ต่อ Lalamove v3 quote API → เทียบ quote จริง (ตามพิกัด) กับสูตรเก็บลูกค้าข้างบน = รู้ว่า subsidize กี่บาท/ออเดอร์ (ที่ประมาณ 30-50K/เดือน) — **ทำไม่ได้จากโค้ดล้วน ต้องมี API key + พิกัดลูกค้าจริงจาก migrate**
3. **ปริมาณ:** จำไม่ได้ ไม่ซ้ำแต่ละวัน · admin จัดการ → **ต้องรอ order data จาก migrate ถึงจะรู้ volume/โซนจริง**
4. **พิกัด:** เหมือนเก็บ lat/lng อยู่ (ยืนยันตอน migrate — โค้ดมี `cusPin`/haversine อยู่แล้ว)
5. **ช่วงเวลาส่ง:** น่าจะมีแล้ว (มี SLOTS: เช้า/บ่าย/รับได้ตลอดวัน)

## 🔴 สถานะ: PARKED หลัง migrate
นัทเคาะ (14 ก.ค.): **พี่เก่งต้องรอ data ลูกค้า+ออเดอร์ทั้งชุดก่อน** (volume/โซน/พิกัด/ประวัติ) → ไปทำ **migrate session** ก่อน แล้วค่อยกลับมาล็อก logic พี่เก่ง · JD โครงเสร็จแล้ว รอ data มาเติมชั้น 1-4 ด้วยตัวเลขจริง

# 💬 น้องเอิธ — Marketing Intelligence & Growth Ops (a0.3-spec · DRAFT · Tier 1)

## 1. Purpose
เป็น "หลังบ้านมาเก็ตติ้ง" ของนัท — เฝ้าตลาด/คู่แข่ง + หาช่องโต + ช่วยยิงแอด + ดูแลลูกค้าเบื้องต้น → feed ข้อมูล/ข้อเสนอให้นัท (และพลอยผ่านนัท) ตัดสินใจ · **codify งานที่ต้องทำสม่ำเสมอแต่ไม่มีคนทำประจำ** (เฝ้าคู่แข่ง วิเคราะห์ตลาด หา influencer)
> ⚠️ **แบ่งเขตกับพลอย:** พลอย = หน้าบ้าน/หน้ากล้อง/ปิดดีล influencer/on-ground · เอิธ = back-office ป้อนข้อมูล+ร่าง+เฝ้า ให้พลอยเอาไปใช้ · **เอิธไม่แย่ง territory หน้าบ้านพลอย**

## 2. Trigger / Cadence (นัทเน้น: weekly + occasionally)

| Google Ads keyword data (ของเก่า: คอร์สอาหารคลีน 127 conv ฯลฯ) | 🟢 มีใน CLAUDE.md |

## 4. Logic / Flow งานหลัก (อัปเดตตามที่นัทเคาะ 15 ก.ค.)
**A. Competitor Watch (weekly):** ไล่ดูคู่แข่ง → จับ "เปลี่ยนอะไร" (เมนู/ราคา/โปร/มุมโฆษณา/persona) → diff + ประเมินภัย → เสนอ counter-move · เฝ้า **ทุกไลน์สินค้าที่ครัวเราทำ** (เพิ่มคู่แข่งไปเรื่อยๆ ไม่จำกัด 4 ราย)
**B. Trend + Market (weekly/monthly):** เก็บเทรนด์อาหาร/ไดเอ็ต/เมนูไวรัล + คำค้นมาแรง → **flow เมนูกระแส: เอิธ→ฟ้า(ทำได้/ต้นทุน)→นิว(จัดผลิต)** + **รับฟีดแบคกลับ** วน loop · วิเคราะห์ช่องไหน (SEO/Ads/Social) คุ้มสุด
**C. Influencer Sourcing (occasionally) — vetting คัด 🍋 lime vs lemon (นัทเน้น):**
  > **หัวใจ:** แยก **lime = เก่งจริง** (มีผลตอบรับจริง คนเชื่อ) ออกจาก **lemon = ดูเก่งแต่ไม่จริง** (ยอด follow สวย/ฟีดสวย แต่ engagement ปั้ม นิสัยไม่ดี จ่ายไปแล้วเงียบ ไม่มี ROI) — เคยเจอ lemon เข้ามา เสียเงินฟรี
  1. หา candidate ตาม niche (สุขภาพ/ฟิตเนส/Hyrox)
  2. **vetting 2 ชั้น:** (ก) **ไลฟ์สไตล์เข้ากับแบรนด์ไหม** (real food/สุขภาพพอดี ไม่ใช่คลีนสุดโต่ง) (ข) **lime หรือ lemon** — เช็ค engagement จริง/ปั้ม · คอมเมนต์คนจริงไหม · เคยรีวิวแล้วมีผลจริงไหม · นิสัย/ความน่าเชื่อถือ
  3. ส่ง shortlist + **เหตุผลที่เลือกนอกเหนือยอด follow** ต่อคน (ทำไมคิดว่าเป็น lime) → **พลอยคัดรอบสุดท้าย + ให้เหตุผลกลับ** (= feedback เทรนเกณฑ์เอิธ ให้แม่นขึ้น)
  - ⚠️ ส่วน vetting นี้ = **sub-module ต้องเทรนแยก** (สะสมเกณฑ์แยก lime/lemon + เคสจริงที่เคยพลาด) — แตกเป็น spec/session ย่อยได้ทีหลัง
**D. Ad Ops — นัทเคาะ: จัดครบจนพลอยแค่กดยิงสุดท้าย (end-to-end prep):**
  - ร่าง copy (ตาม brand) · เลือก audience/targeting · เสนองบ+bidding · negative keywords · brief creative (ส่งต่อเตียงทำภาพ) · เซ็ตโครงแคมเปญให้พร้อม → **พลอยรีวิว + กด publish เอง** (ขั้นกดสุดท้าย = คนทำ)
  - อ่าน metric → เสนอ optimize (นิยาม "อ่านเมตริก" ดูข้อ 4.1)
**E. CS + Review (event-driven):** ตอบพื้นที่ส่ง/ค่าส่งใน LINE (ข้อมูลจริง) · หลังส่งของ → ส่งลิงก์ขอรีวิว (`g.page/r/...`) · เก็บฟีดแบค → เข้าชีท

### 4.1 "อ่านเมตริก" คือทำอะไร (นัทขอขยาย)
เอิธดึงตัวเลขผลแอด แล้ว**แปลเป็นคำแนะนำที่พลอยกดทำได้** ไม่ใช่แค่รายงานเลข:
| เมตริก | อ่านว่า | เอิธเสนออะไร |
|--------|---------|--------------|
| CTR (% คนคลิก) | โฆษณาสะดุดตาไหม | ต่ำ → เปลี่ยน headline/ภาพ |
| CPC (บาท/คลิก) | คลิกแพงไป | สูง → ปรับ bid/keyword |
| Conversion + CPA (บาท/ลูกค้า) | คุ้มไหม | keyword/แอดไหน conv 0 แต่เผาเงิน = **ตัดทิ้ง** · ตัวไหนถูก+conv ดี = **ดันงบเพิ่ม** |
| Search terms | คนพิมพ์อะไรมาเจอเรา | คำมั่ว/ไม่เกี่ยว = เพิ่ม negative keyword |
| Frequency/ยอดใช้จ่าย | จ่ายเกินคุ้มไหม | เกิน = ลด/พัก |
- v1: เอิธอ่านจากตัวเลขที่**นัท/พลอย copy จากคอนโซลมาวาง** (หรือรายงานที่ export) → future ต่อ Google Ads API อ่านเอง
- อิงของจริงที่มี: keyword ทองคำ (คอร์สอาหารคลีน 127 conv ฿87/conv) vs คำเผาเงิน (อาหารสุขภาพ conv 0 เผา ฿1,120) = โมเดลการตัดสินใจของเอิธ

## 5. Output — 📄 ศูนย์กลาง = "ชีทพลอย" (นัทเคาะ)
- **The Sheet (Google Sheet แผ่นเดียว หลาย tab):** พลอยชอบ "กระดาษแผ่นเดียวจบ" ไม่ชอบมาถาม agent → เอิธ maintain ชีทกลาง เปิดปุ๊บใช้ได้เลย · **เขียน task brief ลงชีท** (พลอยเห็นงาน+บรีฟครบ ไม่ต้องแชท)
  - tab เริ่ม (ทยอยเพิ่มไปเรื่อยๆ): `สรุปสัปดาห์` · `Task/Brief` · `Influencer shortlist+เหตุผล` · `Ad ชุดพร้อมยิง` · `คู่แข่งขยับ` · `Feedback/รีวิว`
- **Weekly digest** = tab "สรุปสัปดาห์" บนสุด (ดูนิยามข้อ 5.1)
- **Alert ด่วน** = ช่องแยก/ไฮไลต์ เมื่อเจอเหตุเร่ง
- แสดงที่ไหน: **ชีท = หลัก** (สำหรับพลอย) · อนาคต tab "เอิธ" ใน OH (สำหรับนัท)

## 6. Human-in-loop

- ✅ **Stream A/B/D (คู่แข่ง/เทรนด์/ร่างแอด) เริ่มได้เลย** — ใช้ public data + brand rules ที่มีครบ
- 🟡 Stream C (influencer) ต้องเคาะงบ/เกณฑ์กับพลอยก่อน
- 🔴 Stream E (CS/win-back เชิงลึก) รอ migrate + web_events

## 9. v1 scope vs future
- **v1:** competitor watch + trend feed + weekly digest + ร่าง ad copy/optimize (คนกดคอนโซล) — **ไม่แตะเงิน/ไม่ส่งข้อความนอกเอง**
- **future:** ต่อ Ads API อ่าน metric อัตโนมัติ · influencer CRM เต็ม · CS อัตโนมัติใน LINE OA · win-back segmentation (หลัง migrate)

## 10. ✅ เคาะแล้ว (15 ก.ค.) + ที่ยังค้าง
1. ✅ Ad ops = **v1 ร่าง+เสนอ+อ่านเมตริก (ดูข้อ 4.1)** · จัด end-to-end จนพลอยกด publish ขั้นสุดท้าย
2. ✅ เอิธ vs พลอย = **เอิธจัดทุกอย่างให้ครบ → พลอยกดยิงขั้นสุดท้าย** (พลอยยิงเองอยู่แล้ว เอิธเป็นฝ่ายเตรียม)
3. ✅ Output = **ชีทพลอย (แผ่นเดียว หลาย tab)** + digest = tab สรุปบนสุด (ข้อ 5/5.1)
4. ✅ CS stream E = อยู่กับเอิธก่อน · อาจแตกทีหลังถ้าโหลดเยอะ
5. ✅ คู่แข่ง = **เพิ่มไปเรื่อยๆ ครอบทุกไลน์สินค้าที่ครัวทำ**
